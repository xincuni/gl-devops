# 运维平台功能规划

## 1. 产品定位

这个平台的定位不是替代阿里云、AWS 或 Cloudflare 控制台，而是做一个统一的运维入口，解决四个核心问题：

1. 多云资产分散，缺少统一资产视图
2. DNS 解析分散在不同平台，缺少统一治理能力
3. 云服务器与 JumpServer 资产不一致，纳管链路不完整
4. 云账单难以按业务标签分析，成本可见性不足

一期目标以“统一接入、统一同步、统一展示、统一分析”为主。

## 1.1 实现约束

- 前后端分离
- 后端通过 `Django REST Framework` 提供 API 给前端
- 核心业务流程主要放在前端页面完成
- `Django Admin + Unfold` 主要用于配置、运营和内部管理
- Admin 主题方案固定为 `Unfold`
- `Django Admin` 需要基于 `Unfold` 做主题优化和管理体验优化

## 2. 一期范围

- 云账号管理
- 域名管理
- 服务器管理
- JumpServer 同步

## 3. 业务主线

平台一期围绕三条主线建设：

### 3.1 资产主线

将阿里云 ECS、AWS EC2、DNS 记录、JumpServer 资产统一纳入平台，并沉淀到 CMDB。

### 3.2 同步主线

建立云平台到本地、再从本地到 JumpServer 的同步链路，并形成任务、日志、审计闭环。

### 3.3 成本主线

拉取 AWS 和阿里云账单，以 `env/product` 为核心标签维度做成本分析。

## 4. 模块规划

### 4.1 Portal 首页

定位：统一入口和总览看板。

主要功能：

- 云账号总数
- 域名总数和 DNS 记录总数
- 云主机总数
- JumpServer 同步状态统计
- 本月云费用总览
- 最近任务列表
- 失败任务和异常提示

一期目标：

- 首页先解决“看见什么资产、哪些同步失败、费用大概多少”
- 不在一期做复杂自定义大盘
- 主要业务交互放在独立前端，而不是 Django Admin

### 4.2 云账号管理

定位：所有连接器和同步任务的基础模块。

主要功能：

- 新增阿里云账号
- 新增 AWS 账号
- 新增 Cloudflare 账号
- 新增 JumpServer 连接
- 编辑和停用账号
- 账号连通性测试
- 手动触发首次同步
- 保存最近同步时间和状态

关键要求：

- 凭证加密存储
- 不同 provider 的认证方式可扩展
- 账号维度支持独立启停
- 账号配置可以同时在 `Vue` 前端和 `Django Admin` 后台维护

### 4.3 域名管理

定位：统一管理阿里云和 Cloudflare 的域名与 DNS 记录。

主要功能：

- 同步域名列表
- 同步解析记录
- 解析记录查询和过滤
- 查看记录详情
- 批量启用/停用记录
- 批量修改 TTL
- 导出记录
- DNS 变更审计

查询维度建议：

- provider
- 域名
- 记录类型
- 主机记录
- 状态
- 更新时间

一期重点：

- 先做同步、展示、基础治理
- 高级巡检和风险识别放到后续

### 4.4 服务器管理

定位：统一管理阿里云 ECS 和 AWS EC2 主机资产。

主要功能：

- 同步 ECS
- 同步 EC2
- 主机列表
- 主机详情
- 展示公网 IP、私网 IP、VPC、子网、OS、实例状态
- 展示云账号、区域、标签
- 展示是否已同步到 JumpServer

查询维度建议：

- provider
- account
- region
- instance status
- tag
- JumpServer sync status

### 4.5 JumpServer 同步

定位：将云主机标准化后推送到 JumpServer，形成统一纳管链路。

主要功能：

- 配置同步规则
- 手动同步
- 定时同步
- 已存在资产匹配更新
- 下线资产禁用策略
- 同步日志
- 失败重试

同步规则建议：

- 按标签同步，例如 `env=prod`
- 按账号同步
- 按区域同步
- 仅同步运行中实例

一期重点：

- 先保证“可推送、可更新、可追踪”
- 复杂纳管编排不在一期

### 4.6 CMDB

定位：平台统一资产底座，不只展示云数据，还要形成标准化资产模型。

一期目标：

- 把云主机、DNS 资产、云账号纳入 CMDB
- 建立统一的资产主表和关联关系
- 形成资产唯一标识、来源、状态、标签、更新时间等基础字段

主要功能：

- 资产标准化建模
- 资产来源标记
- 资产状态管理
- 标签归档
- 资产关联关系管理

一期建议纳入的 CMDB 对象：

- 云账号
- 云主机
- 域名
- DNS 记录
- JumpServer 资产映射

二期再考虑：

- 应用
- 服务
- 数据库
- 中间件
- 业务拓扑关系

### 4.7 账单采集

定位：统一采集 AWS 和阿里云账单明细。

主要功能：

- 拉取阿里云账单
- 拉取 AWS 账单
- 原始账单明细入库
- 按月补采
- 定时采集
- 采集日志和失败记录

数据要求：

- 保留原始账单字段
- 保留资源 ID、产品、区域、账号、金额、标签
- 支持账单按月重跑

### 4.8 费用分析

定位：把账单转换成运维和管理能直接看的成本视图。

主要功能：

- 按云厂商分析费用
- 按账号分析费用
- 按月份和日期分析费用
- 按产品分析费用
- 按资源分析费用
- 按标签分析费用
- Top N 成本排行
- 成本趋势图

一期标签主维度：

- `env`
- `product`

一期输出视图建议：

- 本月总费用
- 阿里云 / AWS 费用占比
- 按 `env` 的费用分布
- 按 `product` 的费用分布
- Top 资源成本排行

### 4.9 任务中心

定位：统一承载同步、采集、重试、补采等后台任务。

主要功能：

- 定时任务列表
- 手动任务触发
- 任务执行历史
- 任务状态查看
- 失败重试
- 失败原因记录

任务类型建议：

- DNS 同步
- ECS 同步
- EC2 同步
- JumpServer 同步
- 账单采集
- 账单补采

### 4.10 审计日志

定位：记录关键动作，满足追踪和回溯要求。

主要功能：

- 云账号变更记录
- DNS 记录修改记录
- JumpServer 同步操作记录
- 手动任务执行记录
- 账单规则修改记录

## 5.1 Django Admin 定位

`Django Admin + Unfold` 在一期里不承担主要业务前台，而是承担下面这些后台管理能力：

- 云账号和凭证配置
- 标签规则和同步规则配置
- 任务执行记录查看
- 审计日志查看
- CMDB 基础数据维护

主题与体验要求：

- 基于 `Unfold` 替换默认后台配色和导航样式
- 优化列表页筛选、搜索和详情展示
- 统一品牌名称、Logo、菜单分组
- 提升运维和管理人员日常使用效率

首版建议交付下面这些确定性能力：

### Portal

### 云账号

- 阿里云、AWS、Cloudflare、JumpServer 账号接入

### 域名

- 域名与解析记录同步
- 记录列表
- TTL 批量修改
- 启用/停用

### 主机

- ECS/EC2 同步
- 主机列表和详情

### JumpServer

- 主机推送
- 同步日志

## 7. 功能优先级

### P0

- 云账号管理
- DNS 同步
- ECS/EC2 同步
- 账单采集
- CMDB 基础模型

### P1

- JumpServer 同步
- Portal 首页
- 费用分析
- 审计日志
- 任务中心

### P2

- DNS 差异比对
- 标签规则增强
- 成本异常识别
- CMDB 扩展对象

## 8. 角色规划

建议至少三类角色：

- `Admin`：管理账号、权限、规则、全局配置
- `Ops`：管理域名、主机、同步任务、JumpServer 纳管
- `Finance`：查看账单、费用分析和导出报表

## 9. 当前冻结建议

如果按你现在的修改方向，一期建议冻结为下面这套范围：

1. 接入阿里云、AWS、Cloudflare、JumpServer
2. 建立统一 CMDB 基础模型
3. 完成域名、主机、账单三类数据同步
4. 完成云主机到 JumpServer 的纳管链路
5. 完成按 `env/product` 的成本分析
6. 通过 Portal 首页统一展示资产、任务和费用结果
