# 运维平台开发计划

## 1. 目标

本计划用于把功能规划转成可执行的研发迭代，重点回答三个问题：

1. 每个阶段先做什么
2. 后端、前端、Admin、联调分别做什么
3. 每个迭代完成后如何验收

## 2. 技术栈

- 后端：`Python + Django + Django REST Framework`
- 管理后台：`Django Admin + Unfold`
- 数据库：`MySQL`
- 缓存和消息：`Redis`
- 异步任务：`Celery`
- 定时任务：`Celery Beat`
- 前端：`Vue + Tailwind CSS`
- 部署模式：前后端分离

技术约束：

- `Django REST Framework` 负责对接前端
- 主要业务流程放在 `Vue` 前端
- `Django Admin + Unfold` 主要承担配置、规则、运营、审计查看
- Admin 主题方案固定为 `Unfold`
- `Django Admin` 需要基于 `Unfold` 做主题优化和体验优化

## 3. 实施顺序

建议严格按下面顺序推进：

1. 项目基础骨架
2. 认证、权限、账号、任务底座
3. 域名与 DNS 模块
4. 云主机与 CMDB 基础模型
5. JumpServer 纳管链路
6. 账单采集与费用分析
7. Portal 聚合展示
8. 联调、测试、上线

## 4. 阶段计划

### 阶段 1：需求冻结与设计确认

目标：

- 冻结一期功能范围
- 确认技术栈和系统边界
- 确认模块、页面、API 清单

交付物：

- 功能规划文档
- 技术设计文档
- 模块清单
- 页面清单
- API 清单

完成标准：

- 一期范围不再频繁变化
- 核心模型和接口命名冻结

### 阶段 2：项目基础骨架

目标：

- 初始化 Django 后端
- 初始化 Vue 前端
- 初始化 MySQL、Redis、Celery 基础集成
- 建立 DRF 基础 API 框架
- 建立 `Unfold` Admin 主题基线

交付物：

- 后端基础工程
- 前端基础工程
- 基础配置、日志、异常处理
- `Unfold` 主题基础样式

完成标准：

- 后端可启动
- 前端可启动
- DRF 示例接口可用
- Celery Worker / Beat 可运行
- Admin 完成 `Unfold` 基础品牌化

### 阶段 3：系统底座能力

目标：

- 完成认证和权限
- 完成云账号管理
- 完成任务中心基础能力
- 完成审计日志基础能力

交付物：

- 登录与用户信息接口
- 云账号 CRUD
- 任务记录模型和查询接口
- 审计日志模型和查询接口

完成标准：

- 用户可登录前端
- 可新增云账号并执行连接测试
- 手动触发任务后能看到任务记录
- 写操作能产生日志

### 阶段 4：业务模块交付

目标：

- 完成交付 DNS、主机、CMDB、JumpServer、账单、费用分析

说明：

- 该阶段按迭代推进，见下方详细迭代计划

### 阶段 5：Portal 与上线准备

目标：

- 完成首页聚合展示
- 完成联调、测试、发布准备

交付物：

- Portal 页面
- Portal 聚合接口
- 部署文档
- 验收清单

完成标准：

- 主要流程从前端可闭环
- 主要配置可从 Admin 可维护
- 核心同步链路可在测试环境验证通过

## 5. 迭代计划

### 迭代 1：基础工程与底座

目标：

- 把项目跑起来，并交付第一批可用基础能力

后端任务：

- 初始化 Django 项目和 app 结构
- 集成 DRF、MySQL、Redis、Celery、Celery Beat
- 建立统一配置、日志、异常处理、中间件
- 实现 `auth` 基础接口
- 实现 `accounts` 基础模型和 CRUD
- 实现 `tasks` 基础模型和查询接口
- 实现 `audit` 基础模型和日志写入

前端任务：

- 初始化 Vue 项目、路由、状态管理、请求封装
- 搭建基础 Layout、菜单、登录拦截
- 完成登录页
- 完成云账号列表页基础版本
- 完成任务中心基础列表页

Admin 任务：

- 集成 `Unfold`
- 自定义站点标题、Logo、菜单分组
- 优化云账号、任务记录、审计日志的 Admin 展示
- 建立常用筛选项和搜索项
- 处理 Django `User/Group` 在 `Unfold` 下的样式适配

联调任务：

- 登录接口联调
- 云账号列表、创建、编辑接口联调
- 任务列表接口联调

测试重点：

- 登录流程
- 权限拦截
- 云账号创建和启停
- 审计日志生成

验收标准：

- 用户可登录系统
- 云账号可创建、编辑、启停
- 任务中心能看到基础任务记录
- Admin 可维护云账号和查看日志

### 迭代 2：域名与 DNS

目标：

- 打通阿里云和 Cloudflare DNS 同步链路

后端任务：

- 实现 `providers.aliyun_dns`
- 实现 `providers.cloudflare_dns`
- 完成 `dns_zones`、`dns_records` 模型和迁移
- 实现域名同步任务和解析记录同步任务
- 实现 DNS 记录列表、详情、批量启用/停用、批量修改 TTL、导出接口
- 实现 DNS 变更审计

前端任务：

- 完成域名列表页
- 完成 DNS 记录列表页
- 完成 DNS 记录详情页
- 完成批量操作交互
- 完成导出入口和状态提示

Admin 任务：

- 域名与记录数据的 Admin 管理
- DNS 审计日志筛选和查看
- DNS 同步任务查看

联调任务：

- 域名列表接口联调
- 记录列表和详情接口联调
- TTL 批量修改接口联调
- 批量启用/停用接口联调

测试重点：

- 多 provider 数据标准化
- 记录同步准确性
- 批量修改是否正确回写
- 审计日志是否完整

验收标准：

- 可同步阿里云和 Cloudflare 域名与记录
- 前端可查看记录详情并执行基础治理
- 修改操作可追踪

### 迭代 3：云主机与 CMDB 基础模型

目标：

- 打通 ECS / EC2 同步，并沉淀统一资产模型

后端任务：

- 完成 `cmdb` 基础资产模型

前端任务：

- 完成主机列表页

- 完成 CMDB 资产列表页基础版本
- 完成 CMDB 资产详情页基础版本

Admin 任务：

联调任务：

- 认证与权限
- 云账号管理
- DNS 同步
- ECS/EC2 同步
- CMDB 基础模型
- 账单采集

### P1

- JumpServer 同步
- 费用分析

- 审计日志

### P2

- Admin 深度优化
- CMDB 扩展对象
- 更细的成本治理能力

## 8. 风险点

- 多云 provider 字段差异较大，标准化容易反复调整
- JumpServer 字段映射需要尽早验证，否则会影响后续纳管
- 账单数据量上来后，MySQL 聚合查询会有压力
- DNS 批量修改涉及回写外部平台，必须保证审计和幂等
- `Unfold` 如果只做默认安装，不做菜单和模型页优化，运营体验依旧会很差

## 9. 进入实施前的最后准备

正式开发前建议确认下面几项：

1. Django 项目命名和仓库目录结构
2. 登录认证方案是 `Session` 还是 `Token/JWT`
3. `Unfold` 主题接入方案
4. AWS 账单数据来源是 `CUR` 还是 `Cost Explorer`
5. JumpServer 目标版本和 API 兼容性
